### 概要
URLを渡すとPlaywright + Gemini VisionでスクショをとりAppium/WebView向けテストコードを自動生成するツール

### 技術スタック
TypeScript + Playwright + Dify Cloud

### フェーズ

- V0.1（今日）: URL → スクショ → Gemini Vision解析 → テストコード出力
- V1: 遷移グラフ + 既存E2Eナレッジを加えて精度向上　※Vision解析をDify Cloud（Geminiバックエンド）に切り替え
- V2: 生成コードのAI品質分析・レビュー自動化
- V3: OS別出力分岐（Recorder連携）+ iOS対応
将来: DifyベースのチケットHelperと連携

#### 2026-04-21 設計変更

**テストコード生成方針**
- `codegen.ts`（templateベース）は廃止方向
- AIに直接テストコードを生成させる方式に変更
- セレクタ優先順位は `pw-demo/設計方針.md` に準拠：
  1. `getByRole()`
  2. `getByLabel()` / `getByText()`
  3. `getByTestId()`
  4. CSSセレクタ（最下位）

**DOM取得機能を追加（`src/dom/extractDom.ts`）**
- スクショだけでなくDOMも取得してAIに渡すことで精度向上
- カテゴリ選択式（`--elements` オプション）でDOM取得範囲を絞る
- カテゴリ一覧：
  - ボタン：`button, [role="button"]`
  - 入力フォーム：`input, textarea, select, label`
  - テキスト表示：`h1~h6, p, span, [data-testid]`
  - 画像表示：`img, [role="img"], svg`
  - レイアウト：`header, footer, nav, main, section`
  - リンク：`a[href]`（V1遷移グラフへの布石）
- 取得する要素情報：`tagName`（必須）、`role`、`label`、`testId`（任意）

#### 2026-04-30 設計追記

**pw-demoとの関係**
- pw-demoの実行結果（pages.json・HAR）をDifyナレッジとして再利用する構造
- pw-demoがナレッジを生成するわけではなく、既存の実行結果を活用する

**Difyナレッジ構成**
- `transition-flows.txt`（動的）：pw-demoのpages.jsonから生成。実際に操作したフローの遷移順序
- `page-links.txt`（静的）：訪問済みページのa[href]から取得。遷移できる選択肢一覧
- 性質が異なるため分離して登録する

**精度低下の根本原因（検証で判明）**
- 遷移先の正解をAIが知らないことが主因
- 例：本棚の書影押下 → AIがビューアへの遷移と誤判断（正解は本棚内作品詳細）
- 遷移ナレッジがあれば防げるケース

**HAR取得オプション（未実装・設計メモ）**
- 生成コードに `--har` フラグでHAR記録を付与できる設計を検討
- バグ固有フローのAPIログをpw-demoのAPI比較に追加できる
- 生成コードへの組み込みはオプション扱いとする（主目的はバグ再現コード生成のため）
